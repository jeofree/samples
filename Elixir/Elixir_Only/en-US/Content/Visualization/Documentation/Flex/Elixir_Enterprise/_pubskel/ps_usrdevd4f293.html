Key aspects of a IBM Diagram user interface

To develop a graphical user interface using Diagram, you must be aware of the following three aspects:
  • Skinning
  • Model-driven diagrams
  • Monitoring panels

Skinning

Skins are graphical style elements that are used to control the appearance of a component. The node and link skins are the starting point of a diagram display. Skins can contain multiple elements, such as graphic elements and text, and states, for example, normal or selected. You can create skins to represent trucks, factories, network elements or business process elements. These should have built-in behavior, which means that they react dynamically to data changes or user interaction. A skin is usually written in MXML and applied to a component using MXML or CSS. For more information, see Adobe documentation, Using Flex 4 / Enhancing the user interface / Creating Spark Skins.

Model-driven diagrams

Applications that must automatically create a diagram from a data source (such as workflows, business processes, entity relationships, or network topologies) use a model-driven approach. This kind of display is created by binding the data source elements to nodes and links in the diagram. These nodes and links will be graphically represented using the skins configured according to the data object. For example, you can specify that a given type of data instance is represented by a node with a given skin, and that a particular value of a property is bound to some skin configuration that alters the aspect of the node in the diagram. In most cases, underlying models do not contain any information about the location of the different objects. For example, a business process describes the sequence of different tasks and events, but doesn't contain any information on how to display this sequence. Similarly, a network topology only describes how physical elements are interconnected together. An important aspect of a model-driven display, therefore, is the ability to invoke automatic graph layout algorithms that will compute the optimal placement of the created visual components, including nodes and links. Background images, subgraphs, swimlanes, and interactions such as panning, zooming, and selection, can also be configured.

Monitoring panels

To build custom displays for monitoring or supervisory purposes (such as industrial schematics, business dashboards, and other generic human-machine interfaces), the approach is quite the opposite from the model-driven approach. Start by manually placing nodes on top of a static background. You can then create the graphic representation of the monitoring panel using Diagramcomponents (Graph, Subgraph, Node, and Link) and store the resulting schematic in an MXML file associated with the Flex library containing the skins required to display the graphic objects.
